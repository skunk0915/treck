# 記事作成ルール

## 基本方針

article_policy_sheet.md、character_sheet.mdに基づきarticle_progress.mdの記事を作成して。進捗もarticle_progress.mdで管理。
記事作成時はarticle_policy_sheet.mdに基づき、記事を作成する。
twitterコンテンツ作成時はrule_twitter.mdに基づき、指定された記事を、twitter配信に適した形でtwitterディレクトリに同名ファイルでコンテンツを作成する。
instagramコンテンツ作成時はrule_instagram.mdに基づき、指定された記事を、instagramリールコンテンツに適した形でinstagramディレクトリに同名ファイルでコンテンツを作成する。
youtubeコンテンツ作成時はrule_youtube.mdに基づき、指定された記事を、youtube台本に適した形でyoutubeディレクトリに同名ファイル（CSV形式）でコンテンツを作成する。

## 記事作成フロー

### 1. Web調査
記事作成前にwebからその記事タイトルに関する全ての情報を深く調査し、/summary/フォルダに記事ファイル名と同名でまとめる。

### 2. サマリ作成
各情報にはweb調査時の情報の引用元を記載：
- ページ内の該当箇所の文章一部抜粋（**要約ではなく一字一句正確に抜粋**）
- URL（**トップページではなく該当ページのURL**）

**✅ 重要：出典URLの扱い**
- Web検索APIから取得したURLは、リダイレクトURL（`https://vertexaisearch.cloud.google.com/grounding-api-redirect/...`）を**そのまま使用する**
- URLを短縮したり、別のURLに置き換えたりしてはいけない
- サマリファイルに記載したURLと、記事ファイルに記載するURLは**完全に同一**でなければならない

### 3. 記事作成
記事はそのサマリを元に作成する。サマリの情報は全て記事に反映する。

**会話形式の厳守：**
- 単なる情報の羅列にならないように自然な会話形式の体裁を厳守する
- 情報が充実するほど、JKがただの相槌役になっていて初心者ならではのツッコミとか率直な感想のキャラクターが弱くなりがちなのが悪い
- それは記事としてきれいにまとまりすぎてておもしろくない
- 時には的外れだけど共感できるツッコミが必要
- それが読者側の意見を代弁し、読者を置いてけぼりにしないポイントになる

**出典情報の記載：**
- 記事内の全ての引用箇所に、サマリと同じ出典情報（引用文とURL）を記載する
- URLは**Web検索APIから取得したリダイレクトURLをそのまま使用**する
- `<details><summary>出典</summary>`タグで記載する

### 4. ファイル保存
- 記事ファイル名はサマリファイル名と同じにする
- ファイル保存場所は`/article/`配下

## URL記載ルールの詳細

### ❌ やってはいけないこと
- 短縮URLを使用する
- トップページのURLだけを記載する
- `https://example.com/`のような簡略化されたURLを使用する
- Web検索結果から取得したURLを編集・加工する

### ✅ 正しいURL記載方法
- Web検索APIから返された**完全なリダイレクトURL**をそのまま使用
- 例：`https://vertexaisearch.cloud.google.com/grounding-api-redirect/AUZIYQF80MdmngrcqAoe_m77DXMYiJPk1whekh3Ky041pk...`（長いURLでも省略しない）
- サマリファイルと記事ファイルで**完全に同一のURL**を使用

## チェックリスト

記事完成時に確認：
- [ ] サマリファイルを作成したか
- [ ] Web検索APIから取得した完全なリダイレクトURLを使用しているか
- [ ] サマリと記事で同じURLを使用しているか
- [ ] 引用箇所に一字一句正確な抜粋を記載しているか
- [ ] JKのキャラクターが活きているか（単なる相槌役になっていないか）
- [ ] 会話形式が自然か
- [ ] article_progress.mdを更新したか
